feat: Implement Issue #3 - Asset Data Model & Basic Retrieval

This commit introduces the core data models, validation schemas, business logic, and API endpoints for managing internet-facing assets (domains, subdomains, IPs, services, certificates, technologies).

Key additions and architectural decisions:
- Domain Exceptions: Created `AssetNotFoundError` and `AssetDuplicateError` to keep the deep module interface clean and isolate the API layer from SQLAlchemy database exceptions.
- Database Model: Implemented the `Asset` table using PostgreSQL-specific features like `UUID`, `ARRAY` (for tags), and `JSONB` (for metadata storage) to satisfy complex indexing and storage requirements.
- Strict Pydantic Validation: Refactored `AssetBase` into polymorphic Discriminated Unions (`DomainAsset`, `IpAddressAsset`, `ServiceAsset`, etc.) to map explicitly to `AssetType`. This enforces exact validation rules per asset type without relying on shallow, monolithic if/else blocks, adhering to deep module design principles.
- Service Layer (Deep Module): Created `src/services/asset.py` to abstract all DB creation and retrieval operations, handling deduplication constraints and enforcing data normalization natively.
- Testing Suite: Added fully isolated Pytest integration tests within `tests/api/test_assets.py` that use a `TRUNCATE TABLE` strategy to prevent asynchronous cross-loop state leakage while strictly verifying the HTTP API boundary.
