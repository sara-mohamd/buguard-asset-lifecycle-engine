Architecture Deep-Dive: High-Performance Data Ingestion

This document outlines the architectural decisions and flow of the bulk_import_assets engine. The primary engineering goal of this component is to ingest massive batches of assets from automated scanners with minimal latency and memory overhead.

The Core Challenge: ORM vs. Core

In traditional SQLAlchemy applications, processing a list of incoming items involves querying the database for existing Object Relational Mapper (ORM) models, mutating their state in Python, and committing the session. For batches of 1,000+ assets, this results in:

  • Memory Bloat: Loading thousands of ORM state-tracking objects into RAM.
  • The N+1 Update Problem: The database session emits an individual UPDATE statement for every single modified row upon commit.

To bypass these bottlenecks, we engineered a Fetch-Map-Segregate-Execute pipeline using lightweight SQLAlchemy Core operations.

Execution Stages

1. Lightweight Fetching

We execute a bulk select(Asset.id, Asset.type, Asset.value...) using an OR clause of the incoming composite keys. This retrieves only the primitive data types, skipping ORM instantiation.

2. In-Memory Idempotency (The Merge)

As we loop over incoming assets, existing records are not queried individually. Instead, we mathematically merge the metadata utilizing a recursive Python dictionary deep-merge algorithm (deep_merge_dicts) and perform a Set Union on the tags array. This ensures older nested JSON metadata is preserved while newer fields are updated.

3. Segregation & Mapping

The logic segregates results into two flat Python lists: insert_data (for new assets) and update_data (for existing). Crucially, dictionaries in the update_data list retain their database id. This allows SQLAlchemy to dynamically construct the WHERE id = ... parameter during the bulk update.

4. Single-Roundtrip Batch Execution

Finally, we ship both lists to PostgreSQL in just two round-trips:

await db.execute(update(Asset), update_data)
await db.execute(insert(Asset), insert_data)

This achieves O(1) database queries for the actual writing phase, regardless of the batch size.
